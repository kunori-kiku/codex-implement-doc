# implement-doc Codex Workflow

This directory is a `.codex` home for the `implement-doc` workflow.

The workflow turns a roadmap or implementation document into an auditable Codex implementation run:

- the current Codex session acts as the controller;
- exactly one persistent Codex MCP worker performs implementation work;
- worker prompts, state, reports, incidents, decisions, and notifications are written to disk;
- Resend is used for formal email notification and long reports;
- Telegram is used for short alerts and executable plain-text remote decisions;
- Signal is disabled by default.

## Installed Layout

This repository currently contains:

```text
config.toml
prompts/
  implement-doc.md
skills/
  implement-doc/
    SKILL.md
    assets/
    references/
    scripts/
roadmap.md
```

When the workflow runs inside a project repository, it creates runtime artifacts in that project:

```text
.codex-orchestrator/
  implement-doc/
    state/
    reports/
    incidents/
    prompts/
    decisions/
    remote-decisions/
    notifications/
```

The runtime state file is the source of truth for the run ID, roadmap path, worker thread ID, notification settings, and pending remote decisions.

## What You Must Fill In

There are two setup surfaces:

- `config.toml` keeps the MCP server entries and API credentials used to start those servers.
- `.env` keeps workflow notification routing values that are inconvenient to maintain inside generated run state.

Put `.env` next to `config.toml` in the Codex home you actually use. For this repository-as-home setup that is `/home/kunorikiku/source/codex-workflow-deploy/.env`; for a normal install that is `~/.codex/.env`.

In `config.toml`, fill the MCP credential placeholders:

```toml
[mcp_servers.telegram_notifier.env]
TELEGRAM_BOT_TOKEN = "FILL_ME_TELEGRAM_BOT_TOKEN"
TELEGRAM_CHAT_ID = "FILL_ME_TELEGRAM_CHAT_ID"

[mcp_servers.resend.env]
RESEND_API_KEY = "FILL_ME_RESEND_API_KEY"
```

Then copy the workflow notification example:

```bash
cp .env.example .env
```

Fill `.env`:

```env
TELEGRAM_USER_IDS=FILL_ME_TELEGRAM_USER_ID

SENDER_EMAIL_ADDRESS=Codex Orchestrator <alerts@your-verified-domain.com>
RESEND_TO=you@example.com
RESEND_CC=
RESEND_BCC=
REPLY_TO_EMAIL_ADDRESSES=you@example.com
```

Meaning:

```text
TELEGRAM_USER_IDS         Comma-separated Telegram user IDs allowed to approve/reject/redirect runs.
SENDER_EMAIL_ADDRESS      Verified Resend sender address.
RESEND_TO                 Comma-separated email recipients for incident reports and long notifications.
RESEND_CC                 Optional comma-separated CC recipients.
RESEND_BCC                Optional comma-separated BCC recipients.
REPLY_TO_EMAIL_ADDRESSES  Optional reply-to address. Email replies are not executable approvals by default.
```

`config.toml` starts Resend and Telegram directly through `npx`; there is no wrapper entrypoint.

`init_implement_doc_run.py` reads `.env` when it creates a new `.codex-orchestrator/implement-doc/state/*.json` file, then copies these workflow notification values into that generated state.

If a run already exists, it keeps its existing generated state. Start a new run or update that existing state manually if you want old runs to use newly changed notification settings.

## Getting Telegram Token and IDs

Telegram does not give you one generic API key for this workflow. You need three values:

```text
TELEGRAM_BOT_TOKEN  = token for the bot that sends and reads messages; put this in config.toml
TELEGRAM_CHAT_ID    = chat, group, or channel where notifications are sent; put this in config.toml
TELEGRAM_USER_IDS   = user IDs allowed to approve or redirect runs; put this in .env
```

Official Telegram documentation says bot tokens are generated by `@BotFather`, and the Bot API uses `getUpdates` to receive incoming updates.

### Create the Bot Token

1. Open Telegram.
2. Start a chat with `@BotFather`.
3. Send:

```text
/newbot
```

4. Follow the prompts to choose a display name and username.
5. Copy the bot token. It looks like:

```text
123456789:AAExampleToken...
```

6. Put it in `config.toml`:

```toml
TELEGRAM_BOT_TOKEN = "123456789:AAExampleToken..."
```

### Get the Chat ID

1. Start a chat with your bot, or add the bot to the group/channel where you want notifications.
2. Send any message to the bot or to that chat.
3. Call `getUpdates`:

```bash
curl "https://api.telegram.org/bot<TELEGRAM_BOT_TOKEN>/getUpdates"
```

4. Find the `chat.id` field:

```json
"chat": {
  "id": 123456789
}
```

Use that value in `config.toml`:

```toml
TELEGRAM_CHAT_ID = "123456789"
```

For groups and supergroups, chat IDs are often negative, for example:

```text
-1001234567890
```

Keep the value exactly as Telegram reports it.

### Get Your Telegram User ID for the Allowlist

The workflow also needs to know who is allowed to approve or redirect a run. This is not always the same as the chat ID.

In the same `getUpdates` response, find the `from.id` field for your own message:

```json
"from": {
  "id": 123456789
}
```

Use that value in `.env`:

```env
TELEGRAM_USER_IDS=123456789
```

For multiple allowed users, use commas:

```env
TELEGRAM_USER_IDS=123456789,987654321
```

### Common Telegram Failure Cases

- If `getUpdates` returns an empty array, send a new message to the bot and try again.
- If your bot is in a group, privacy mode may affect which messages it sees. For a simple setup, test in a direct chat first.
- If you use a group or supergroup, use the `chat.id`, not your personal `from.id`, for `TELEGRAM_CHAT_ID`.
- Use your personal `from.id` for `TELEGRAM_USER_IDS`.

## Getting Resend API Key and Sender Setup

Resend needs more than just an API key. For this workflow you need:

```text
RESEND_API_KEY          = Resend API key used by the MCP server; put this in config.toml
SENDER_EMAIL_ADDRESS    = verified sender address used in outgoing email; put this in .env
recipient email address = where Codex sends incident reports; put this in .env
```

You usually also need a verified sending domain before Resend will send production email reliably.

### Verify a Sending Domain

1. Create or log into a Resend account.
2. Open the Resend dashboard.
3. Go to Domains.
4. Add a domain.
5. Prefer a subdomain such as:

```text
alerts.example.com
```

6. Resend will show DNS records to add at your DNS provider.
7. Add the DNS records exactly as shown. Resend documents SPF and DKIM as required for domain verification.
8. Wait until the domain status is verified in Resend.

### Create the Resend API Key

1. Open the Resend dashboard.
2. Go to API Keys.
3. Create a new API key.
4. Prefer `Sending access` instead of `Full access`.
5. If Resend offers domain scoping, restrict the key to your notification domain.
6. Copy the key once and store it securely. Resend API keys are secrets.
7. Put it in `config.toml`:

```toml
RESEND_API_KEY = "re_..."
```

### Configure the Sender Address

After a domain is verified in Resend, Resend allows sending from addresses at that domain. The address does not have to be a real inbox, but using a reply-capable address is recommended.

Put the sender and optional reply-to address in `.env`:

```env
SENDER_EMAIL_ADDRESS=Codex Orchestrator <alerts@alerts.example.com>
REPLY_TO_EMAIL_ADDRESSES=you@example.com
```

`REPLY_TO_EMAIL_ADDRESSES` is optional. The workflow still treats email as notification-only by default.

### Configure Recipient Email

The MCP server can send email only after the workflow provides a recipient. Put recipients in `.env`:

```env
RESEND_TO=you@example.com
RESEND_CC=
RESEND_BCC=
```

For multiple recipients, use commas:

```env
RESEND_TO=owner@example.com,backup@example.com
```

New run state files inherit these values automatically. If a run already exists, it keeps its existing generated state; start a new run or update that state manually.

### Minimal Resend Checklist

Before testing real Resend email, confirm:

- `RESEND_API_KEY` starts with `re_`;
- the sending domain is verified in Resend;
- `SENDER_EMAIL_ADDRESS` uses that verified domain;
- `RESEND_TO` contains at least one recipient email;
- the recipient mailbox can receive external email.

## Operational Workflow

1. Open a project repository that contains an implementation document, roadmap, SPEC, TODO plan, or similar document.
2. Start Codex from the project root with this `.codex` home:

```bash
CODEX_HOME=/home/kunorikiku/source/codex-workflow-deploy codex
```

3. Invoke the slash prompt:

```text
/prompts:implement-doc DOC=docs/roadmap.md TITLE="my project implementation"
```

Or invoke the skill directly:

```text
$implement-doc

Implement as per the doc under docs/roadmap.md says.

Title: my project implementation

Use the current repository root.
```

4. The controller reads the roadmap and creates or resumes a run state under:

```text
.codex-orchestrator/implement-doc/state/
```

5. The controller assigns one bounded task to the persistent worker.
6. The worker writes code and writes a report under:

```text
.codex-orchestrator/implement-doc/reports/
```

7. The controller performs lightweight verification:

```bash
git status --short
git diff --name-status
git diff --stat
git diff --check
```

8. If the result is safe, the controller continues with the next bounded task.
9. If there is a serious issue, the controller writes an incident report and notification artifacts.
10. For SEV0-SEV2 issues, Resend sends the long report by email and Telegram sends a short decision prompt.
11. You reply in Telegram using the required plain-text grammar.

## Telegram Remote Decision Format

Every executable reply must include the nonce shown in the notification:

```text
A <nonce>
B <nonce>
C <nonce>
D <nonce>
E <nonce>
F <nonce>
CUSTOM <nonce> <freeform instruction>
```

Meanings:

```text
A      Continue current route and update roadmap later.
B      Revert worker changes for this task.
C      Keep partial changes but redirect implementation.
D      Pause; user will provide a new roadmap section.
E      Stop this implementation run.
F      Run a read-only verifier audit first.
CUSTOM Follow a custom controller instruction.
```

Example:

```text
C 8KQ2
```

Custom example:

```text
CUSTOM 8KQ2 keep the GPU filtering changes, but revert public solver API changes
```

The controller must validate:

- sender Telegram user ID is allowlisted;
- nonce matches the pending incident;
- reply format is supported;
- the decision is written to `.codex-orchestrator/implement-doc/remote-decisions/`.

## Safety Rules

- Do not expose `codex_worker` to the worker profile if your Codex setup supports profile-level MCP isolation.
- Do not use `danger-full-access` for worker sessions unless the environment is disposable and you explicitly accept that risk.
- Do not use Resend inbound email as executable approval by default.
- Do not use Telegram inline keyboards by default.
- Do not use Signal by default.
- Do not continue a SEV0-SEV2 run without a valid Telegram decision, direct user instruction, or roadmap update.
- Keep API keys out of commits and logs.
- `config.toml` contains API-key placeholders by default. If you publish a fork after filling real credentials, remove the real values first.

## Troubleshooting

If `codex mcp list` does not show the servers, confirm:

```bash
echo "$CODEX_HOME"
```

and verify that `config.toml` is inside that directory.

If Telegram starts but messages do not arrive:

- verify `TELEGRAM_BOT_TOKEN` in `config.toml`;
- verify `TELEGRAM_CHAT_ID` in `config.toml`;
- send a message to the bot first;
- for group chats, confirm the bot has been added to the group;
- use `getUpdates` to confirm the reported chat ID.

If Resend starts but email fails:

- verify `RESEND_API_KEY` in `config.toml`;
- verify the sender domain in Resend;
- confirm SPF and DKIM records are verified;
- confirm `SENDER_EMAIL_ADDRESS` in `.env` uses the verified domain;
- confirm `RESEND_TO` in `.env` is not empty.

If `triage_diff.sh` says it is not inside a Git worktree, run it from the project repository, not from this `.codex` home.
